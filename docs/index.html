n8n Self-Healing

Code-first n8n workflows that demonstrate OpenRouter-backed diagnosis, retry, fallback, escalation, and simulator-driven failure testing using n8nac.

Architecture

The repo packages three workflows: a primary sync pipeline, a self-healer workflow, and an error simulator that deliberately feeds recoverable failures into the healer.

graph TD A["POST /webhook/api-data-sync"] --> B["Fetch Source Posts"] B --> C["Transform Source Data"] C --> D["Persist Snapshot to Static Data"] B -. error .-> E["Build Error Context"] C -. error .-> E D -. error .-> E E --> F["POST /webhook/self-healer"] G["POST /webhook/simulate-error"] --> H["Generate 429 / 500 / 401 / parse / timeout / schema"] H --> F F --> I["Diagnose"] I --> J["Retry / Backoff / Fallback / Escalate"]

Workflows

API Data Sync

Fetches JSONPlaceholder data, transforms records, stores the latest snapshot in workflow static data, and forwards recoverable failures to the healer.

Self-Healer

Chooses retry, backoff, fallback, or escalation, records heal history in workflow static data, and returns a structured healing result.

Error Generator

Simulates multiple failure classes so the repair flow can be demonstrated repeatedly without depending on random upstream errors.

Quick Start

npm install
git config core.hooksPath .githooks
bd init        # only for a fresh copy that does not already include .beads/
bd prime
export N8N_API_KEY="<your n8n API key>"
export OPENROUTER_API_KEY="<your openrouter key>"
export SLACK_WEBHOOK_URL="https://hooks.slack.com/services/..."
npm run setup:n8n -- http://172.31.224.1:5678
npm run validate:workflows
npm run validate

npx --yes n8nac push /absolute/path/to/self-healer.workflow.ts --verify
npx --yes n8nac push /absolute/path/to/api-data-sync.workflow.ts --verify
npx --yes n8nac push /absolute/path/to/error-generator.workflow.ts --verify

Repository Conventions

  • `workflows/` contains the project workflow packages.
  • `template/` provides the scaffold used by `npm run new-workflow`.
  • `workflow/` is the root-level single-workflow export slot for distribution.
  • `docs/` is reserved for lightweight Pages-style project documentation.
